This document defines the fundamental rules and workflow for creating prototypes within our design ecosystem.

---

## 01. DEVELOPMENT STAGES & VARIABILITY (HIGHEST PRIORITY)

Before starting the creation of final interfaces, a sequential structural approval workflow must be applied:
1. **Hero Sections (Low-Fidelity / Conceptual Layouts):** When defining the design, you should first create several distinct **Hero sections**. This allows the team to quickly test composition, interface logic, and determine the final design direction without being distracted by color accents or secondary elements.
2. **Full Prototypes (High-Fidelity):** Only after the core design and structure of the Hero sections have been officially approved are you permitted to proceed with creating full, detailed page prototypes.

---

## 02. TECHNICAL & ARCHITECTURAL RULES

### Single Source of Truth for Styles
- **Single CSS File:** The entire prototype must utilize **only one external CSS file** (e.g., `style.css`).
- Splitting styles across multiple files or overusing inline styles in the HTML markup is strictly prohibited. This single file will be reused to maintain system integrity.

### Cataloging & Navigation
- **Prototypes Catalog:** The root directory of the project must contain a main HTML file (e.g., `index.html`) that serves as an interactive hub or site map.
- This file must provide clear and user-friendly navigation linking to absolutely all created page prototypes for quick review by the client or team.

---

## 03. QUALIFICATION & EXECUTION QUALITY

### Professional Standard
- **UIUX Pro Max Skill:** UI UX Pro Max skill must be applied when designing complex prototypes or dashboards.
